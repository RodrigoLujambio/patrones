⛔ Antipatrones

Blob · Lava Flow · Golden Hammer · Spaghetti · Cut&Paste · TDD

¿Qué es un antipatrón?

Una solución habitual a un problema recurrente que parece beneficiosa a corto plazo, pero que produce resultados contraproducentes a largo plazo.

⚠️ Lo que hace a una práctica un antipatrón no es solo ser una mala solución, sino parecer buena en la superficie y traer problemas en el futuro. Suelen surgir de experiencia limitada, desconocimiento de buenas prácticas o plazos apresurados.

¿Cómo se reconocen?

🐢 Ineficiencia

Hacen el software más lento o le hacen consumir más recursos de los necesarios.

🔧 Difícil de mantener

Vuelven el código más difícil de entender, modificar o extender.

🐛 Introducen defectos

Hacen más probable la aparición de errores.

🌀 Complejidad innecesaria

Agregan complejidad al diseño sin un beneficio claro.

🫧 The Blob

Una sola clase concentra casi todas las responsabilidades, mientras las demás son meros contenedores de datos: programación procedural disfrazada de orientación a objetos.

🚩 Cómo reconocerlo

  • Una clase con muchísimos atributos y métodos.
  • Baja cohesión entre sus atributos y operaciones.
  • Las demás clases solo guardan datos, con poca o nula lógica.

🛠️ Cómo evitarlo

  • Dividir la clase gigante en clases pequeñas y especializadas.
  • Redistribuir responsabilidades aplicando SRP.
  • Que cada clase tenga un propósito claro y único.

🌋 Lava Flow

Piezas de código que quedan en el sistema aunque ya no son útiles y que se vuelven difíciles de eliminar (código que “se endurece” como la lava).

🚩 Cómo reconocerlo

  • Código obsoleto que sigue ahí “por las dudas”.
  • Código que ya nadie entiende ni documenta.
  • Código que se evita tocar por miedo a romper algo.

🛠️ Cómo evitarlo

  • Documentar bien el código y sus cambios.
  • Hacer revisiones de código periódicas.
  • Refactorizar y eliminar código muerto tras pruebas rigurosas.

🔨 Golden Hammer

“Si solo tengo un martillo, todo parece un clavo”: depender demasiado de una herramienta o tecnología familiar, a costa de mejores alternativas.

🚩 Cómo reconocerlo

  • Sobreutilización de una herramienta o tecnología.
  • Reticencia a considerar o adoptar alternativas.
  • Creer que la herramienta familiar resuelve todo.

🛠️ Cómo evitarlo

  • Capacitar al equipo en distintas herramientas y tecnologías.
  • Evaluar opciones objetivamente antes de elegir.
  • Fomentar la experimentación y el aprendizaje continuo.

🍝 Spaghetti Code

Código con una estructura enredada y compleja —como un plato de espaguetis— difícil de leer, mantener y depurar.

🚩 Cómo reconocerlo

  • Falta de estructura y organización.
  • Flujo de control difícil de seguir.
  • Ausencia de modularidad y encapsulación.

🛠️ Cómo evitarlo

  • Diseñar la arquitectura antes de escribir el código.
  • Seguir buenas prácticas y hacer revisiones de código.
  • Dividir en funciones y clases pequeñas y coherentes.

📋 Cut-and-Paste Programming

Copiar y pegar bloques de código por toda la aplicación en lugar de crear funciones o módulos reutilizables.

🚩 Cómo reconocerlo

  • Código duplicado por todos lados.
  • Falta de modularidad y abstracción.
  • Un cambio hay que replicarlo en muchos lugares.

🛠️ Cómo evitarlo

  • Pensar en modularidad y reutilización desde el inicio.
  • Extraer funciones y clases para la lógica común.
  • Identificar y eliminar la duplicación (principio DRY).

🐛 Tester Driven Development

También llamado “Bug Driven Development”: que las pruebas y los reportes de errores dirijan el desarrollo, en lugar de las necesidades del usuario y los requerimientos.

🚩 Cuándo ocurre

  • Las pruebas empiezan demasiado pronto.
  • Los requerimientos están incompletos.
  • Testers/desarrolladores inexpertos o gestión deficiente.

🛠️ Cómo evitarlo

  • Empezar a probar en el momento adecuado.
  • Definir requerimientos completos y claros.
  • Mantener el foco en el valor para el usuario.

🧭 Conocer los antipatrones nos dice qué evitar. Los patrones de diseño nos dan el camino contrario: soluciones probadas a problemas recurrentes.

¿Qué son los patrones de diseño?

Son soluciones probadas y reutilizables a problemas recurrentes del diseño de software. No son código que se copia y pega, sino ideas y estructuras que se adaptan a cada situación.

📘 Se popularizaron en 1994 con el libro “Design Patterns: Elements of Reusable Object-Oriented Software”, escrito por los cuatro autores conocidos como “Gang of Four” (GoF).

Las 3 familias de patrones

🏗️ Creacionales

Controlan cómo se crean los objetos, separando la instanciación de la lógica.

Singleton · Factory Method · Builder · Prototype

🧱 Estructurales

Organizan clases y objetos en estructuras flexibles y reutilizables.

Adapter · Bridge · Decorator · Facade · Proxy

🔄 De comportamiento

Definen cómo interactúan los objetos y cómo se reparten las responsabilidades.

CoR · Command · Observer · Mediator · State · Strategy · …

¿Por qué usarlos?

  • Aportan un vocabulario común: decir “acá uso un Observer” comunica una idea completa en dos palabras.
  • Capturan buenas prácticas orientadas a objetos (bajo acoplamiento, alta cohesión, principios SOLID).
  • Hacen el diseño más flexible, extensible y mantenible frente al cambio.
⚠️ No son una bala de plata: aplicarlos sin necesidad agrega complejidad. El patrón resuelve un problema, no debe inventarlo.

🏗️ Patrones Creacionales

Singleton · Factory Method · Builder · Prototype

Los patrones creacionales abordan la creación de objetos: garantizar una única instancia, crear familias de objetos de forma flexible o construir objetos complejos paso a paso, manteniendo el código desacoplado de las clases concretas.

Creacional

🔒 Singleton

Una única instancia, un punto de acceso global.

¿Qué es?

Garantiza que una clase tenga una sola instancia en todo el programa y provee un punto de acceso global a ella. La instancia se crea solo cuando se necesita por primera vez (inicialización perezosa).

Problemas que resuelve · y sus costos

✅ Resuelve

  • Asegura una única instancia de la clase.
  • Da un punto de acceso global y controlado.
  • Inicializa el recurso solo al primer uso.

⚠️ Cuidado

  • Viola el SRP: crea instancias y las limita.
  • Introduce estado global → acoplamiento oculto.
  • Complica los tests (difícil de aislar/mockear).

Estructura (UML)

Implementación


class AppConfig {
  // instancia única (privada y estática)
  static #instance = null;
  #settings = {};

  // el constructor se mantiene "interno":
  // siempre se accede a la clase vía getInstance()
  static getInstance() {
    if (!AppConfig.#instance) {
      AppConfig.#instance = new AppConfig();
    }
    return AppConfig.#instance;
  }

  set(key, value) { this.#settings[key] = value; }
  get(key) { return this.#settings[key]; }
}

Uso desde el cliente


const a = AppConfig.getInstance();
a.set("theme", "dark");

const b = AppConfig.getInstance();
console.log(b.get("theme")); // "dark"
console.log(a === b);        // true  → es la MISMA instancia

Ejemplos reales 🌍

🖨️ Cola de impresión

Una sola instancia coordina los trabajos y evita conflictos entre impresiones simultáneas.

🎮 Motor de videojuego

El núcleo que gestiona el ciclo de renderizado debe ser único para evitar inconsistencias.

Creacional

🏭 Factory Method

Delegá la creación de objetos a las subclases.

¿Qué es?

Define una interfaz para crear un objeto, pero deja que las subclases decidan qué clase concreta instanciar. Separa el código que usa el producto del código que lo construye.

¿Cuándo usarlo? 🧠

  • Cuando trabajás con jerarquías de clases y necesitás crear objetos polimórficos.
  • Cuando querés desacoplar el cliente de las clases concretas.
  • Cuando querés extender el código agregando nuevas subclases sin tocar lo existente (OCP).

Estructura (UML)

Producto e interfaz


// Producto: interfaz común a todos los botones
class Button {
  render() { throw new Error("No implementado"); }
  onClick(handler) { throw new Error("No implementado"); }
}

class WindowsButton extends Button {
  render() { console.log("🪟 Botón estilo Windows"); }
  onClick(handler) { /* ... */ }
}

class HTMLButton extends Button {
  render() { console.log("🌐 Botón estilo HTML"); }
  onClick(handler) { /* ... */ }
}

Creator con el Factory Method


class Dialog {
  // Factory Method: cada subclase decide el producto concreto
  createButton() { throw new Error("No implementado"); }

  // lógica común que NO depende de la clase concreta
  render() {
    const okButton = this.createButton();
    okButton.onClick(() => console.log("cerrar diálogo"));
    okButton.render();
  }
}

class WindowsDialog extends Dialog {
  createButton() { return new WindowsButton(); }
}

class WebDialog extends Dialog {
  createButton() { return new HTMLButton(); }
}

Uso desde el cliente


function crearDialog(plataforma) {
  return plataforma === "windows" ? new WindowsDialog()
                                  : new WebDialog();
}

const dialog = crearDialog("windows");
dialog.render(); // 🪟 Botón estilo Windows

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • Evita el acoplamiento entre el creador y los productos.
  • Aplica los principios OCP y SRP.
  • Centraliza la creación en un solo lugar.

⚠️ Desventajas

  • Aumenta la cantidad de clases del diseño.
  • Puede ser excesivo para jerarquías pequeñas.

Ejemplos reales 🌍

🖱️ Botones multiplataforma

Una ButtonFactory genera WindowsButton, MacButton o LinuxButton sin cambiar el cliente.

💳 Medios de pago

Una PaymentFactory entrega CreditCardPayment, CryptoPayment, etc. según la elección del usuario.

Creacional

👷 Builder

Construí objetos complejos paso a paso.

¿Qué es?

Permite construir objetos complejos paso a paso, separando la lógica de construcción de la representación final. Con el mismo proceso se pueden producir distintas variantes del objeto.

¿Cuándo usarlo? 🧠

  • Cuando un objeto requiere una inicialización laboriosa, con muchos campos y objetos anidados.
  • Cuando querés evitar constructores con una larga lista de parámetros.
  • Cuando varias representaciones de un producto comparten pasos de construcción similares.

Estructura (UML)

Producto y Builder


// Producto complejo
class Car {
  constructor() {
    this.seats = 0;
    this.engine = null;
    this.tripComputer = false;
    this.gps = false;
  }
}

// Builder: cada "step" devuelve this para encadenar llamadas
class CarBuilder {
  constructor() { this.reset(); }
  reset()              { this.car = new Car();        return this; }
  setSeats(n)          { this.car.seats = n;          return this; }
  setEngine(engine)    { this.car.engine = engine;    return this; }
  setTripComputer()    { this.car.tripComputer = true; return this; }
  setGPS()             { this.car.gps = true;         return this; }
  getResult()          { return this.car; }
}

Director y uso


// Director: conoce la "receta" de cada configuración
class Director {
  makeSportsCar(builder) {
    return builder
      .reset()
      .setSeats(2)
      .setEngine("Sport V8")
      .setTripComputer()
      .setGPS()
      .getResult();
  }
}

const director = new Director();
const sportsCar = director.makeSportsCar(new CarBuilder());

console.log(sportsCar);
// Car { seats: 2, engine: 'Sport V8', tripComputer: true, gps: true }

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • Construye objetos paso a paso; permite aplazar pasos.
  • Reutiliza el mismo código para varias representaciones.
  • Aísla la lógica de construcción (SRP).

⚠️ Desventajas

  • Aumenta la complejidad: exige clases nuevas.
  • Puede ser innecesario para objetos simples.

Ejemplos reales 🌍

📄 Generador de PDF

Portada, índice, secciones, gráficos y pies de página: el builder arma el documento sin que el cliente toque el objeto PDF.

🛡️ Póliza de seguros

Cobertura, deducibles, beneficiarios y cláusulas: el builder arma la póliza según el perfil del cliente.

Creacional

🧬 Prototype

Creá objetos nuevos clonando uno existente.

¿Qué es?

Crea objetos nuevos clonando un objeto existente (el “prototipo”), en lugar de construirlos desde cero. Permite copiar objetos sin acoplarse a sus clases concretas.

¿Cuándo usarlo? 🧠

  • Cuando querés evitar depender de las clases concretas de los objetos a copiar.
  • Cuando crear el objeto desde cero es costoso y clonar es más eficiente.
  • Cuando necesitás gestionar variantes o preajustes de un “mismo” objeto.

Clonación superficial vs. profunda

🪶 Superficial (shallow)

Copia las referencias: el clon y el original comparten los objetos anidados.

🌳 Profunda (deep)

Copia también los objetos referenciados: el clon es totalmente independiente.

Estructura (UML)

Implementación


class Shape {
  constructor(x, y, color) {
    this.x = x; this.y = y; this.color = color;
  }
  clone() { throw new Error("No implementado"); }
}

class Circle extends Shape {
  constructor(x, y, color, radius) {
    super(x, y, color);
    this.radius = radius;
  }
  clone() {
    // copia el estado en un objeto NUEVO del mismo tipo
    return new Circle(this.x, this.y, this.color, this.radius);
  }
}

Uso desde el cliente


const original = new Circle(10, 20, "rojo", 5);
const copia = original.clone();

copia.x = 99;

console.log(original.x);        // 10  → el original no se ve afectado
console.log(copia.x);           // 99
console.log(original !== copia); // true → son objetos distintos

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • Evita reinicializar objetos clonando prototipos prearmados.
  • Crea objetos complejos sin acoplarse a clases concretas.
  • Alternativa a la herencia para manejar preajustes.

⚠️ Desventajas

  • Clonar objetos con referencias circulares puede ser complicado.
  • La clonación profunda requiere cuidado extra.

Ejemplos reales 🌍

📑 Contratos estándar

Un contrato base se clona y luego se personaliza para cada cliente: base.clone() + datos del cliente.

🎮 Animaciones

Se define una animación base (correr, saltar) y se clona con distintos tiempos, colores o velocidades.

🧱 Patrones Estructurales

Adapter · Bridge · Decorator · Facade · Proxy

Los patrones estructurales se ocupan de cómo se componen clases y objetos para formar estructuras más grandes, flexibles y reutilizables, sin que la composición se vuelva rígida.

Estructural

🔌 Adapter

Hacé que dos interfaces incompatibles trabajen juntas.

¿Qué es?

🔧 Permite que dos interfaces incompatibles trabajen juntas. Actúa como un puente entre dos clases que no podrían interactuar directamente por diferencias en sus interfaces.

¿Cuándo usarlo? 🧠

  • Cuando querés usar una clase existente pero su interfaz no coincide con lo que espera tu código.
  • Cuando necesitás integrar código heredado o de terceros sin modificarlo.

Componentes principales 🧩

Target

La interfaz que espera el cliente.

Adaptee

La clase existente con la funcionalidad deseada.

Adapter

Traduce la interfaz del Adaptee a la del Target.

Target y Adaptee


// Target: interfaz esperada por el cliente
class Printer {
  printMessage(msg) { throw new Error("Debe implementarse"); }
}

// Adaptee: clase ya existente, con otra interfaz
class LegacyPrinter {
  oldPrint(msg) { console.log(">>> " + msg); }
}

Adapter y uso


// Adapter: adapta LegacyPrinter a la interfaz Printer
class PrinterAdapter extends Printer {
  constructor() {
    super();
    this.legacy = new LegacyPrinter();
  }
  printMessage(msg) {
    this.legacy.oldPrint(msg); // traduce la llamada
  }
}

// El cliente usa el adapter sin conocer LegacyPrinter
const printer = new PrinterAdapter();
printer.printMessage("Hola mundo"); // >>> Hola mundo

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • Reutiliza código existente sin modificarlo.
  • Aísla al cliente de cambios en la implementación.
  • Facilita integrar librerías o sistemas legados.

⚠️ Desventajas

  • Agrega complejidad si se abusa.
  • Puede implicar capas extra de herencia o composición.

Ejemplos reales 🌍

Estructural

🌉 Bridge

Separá una abstracción de su implementación.

¿Qué es?

🧱 Separa una abstracción de su implementación, de modo que ambas puedan evolucionar de forma independiente.

¿Cuándo usarlo? 🧠

  • Cuando una jerarquía de clases podría crecer en múltiples dimensiones.
  • Cuando querés combinar libremente una abstracción con distintas implementaciones.

Componentes principales 🧩

Abstraction

La interfaz de alto nivel para el cliente.

RefinedAbstraction

Extiende la abstracción y delega en un implementor.

Implementor

Define la interfaz de implementación.

ConcreteImplementor

Implementaciones concretas del implementor.

Abstraction y RefinedAbstraction


// Abstraction: mantiene una referencia al implementor
class Shape {
  constructor(drawingAPI) { this.drawingAPI = drawingAPI; }
  draw() { throw new Error("Debe implementarse"); }
}

class Circle extends Shape {
  constructor(x, y, radius, drawingAPI) {
    super(drawingAPI);
    this.x = x; this.y = y; this.radius = radius;
  }
  draw() { this.drawingAPI.drawCircle(this.x, this.y, this.radius); }
}

Implementor y ConcreteImplementors


// Implementor: interfaz de bajo nivel
class DrawingAPI {
  drawCircle(x, y, radius) { throw new Error("Debe implementarse"); }
}

class CanvasAPI extends DrawingAPI {
  drawCircle(x, y, radius) {
    console.log(`Canvas: círculo en (${x}, ${y}) r=${radius}`);
  }
}

class SVGAPI extends DrawingAPI {
  drawCircle(x, y, radius) {
    console.log(`<circle cx='${x}' cy='${y}' r='${radius}' />`);
  }
}

Uso desde el cliente


// La MISMA forma, con distintas implementaciones de render
const canvasCircle = new Circle(10, 20, 5, new CanvasAPI());
canvasCircle.draw(); // Canvas: círculo en (10, 20) r=5

const svgCircle = new Circle(0, 0, 10, new SVGAPI());
svgCircle.draw();    //

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • Desacopla abstracción e implementación.
  • Permite combinarlas libremente (composición).
  • Facilita extender sin tocar lo existente.

⚠️ Desventajas

  • Introduce más clases y capas.
  • Innecesario si la jerarquía es pequeña.

Ejemplos reales 🌍

📱 Google Pay

Billetera digital que paga con una tarjeta sin entregarla físicamente.

Abstracción: la forma de pagar (app móvil, POS, smartwatch).
Implementación: el medio de pago real (Visa, MasterCard).

✈️ LATAM Airlines

La experiencia de reserva es única, pero la lógica varía por país.

Abstracción: reserva general (búsqueda, selección, check-in).
Implementación: lógica por país (monedas, impuestos, medios de pago, validación de identidad).

🧠 La Abstracción es una capa de control de alto nivel que delega el trabajo real a la capa de implementación.
Estructural

🎁 Decorator

Agregá responsabilidades a un objeto en tiempo de ejecución.

¿Qué es?

✨ Permite agregar responsabilidades a un objeto de forma dinámica, envolviéndolo, sin modificar su código ni crear subclases para cada combinación.

¿Cuándo usarlo? 🧠

  • Cuando querés agregar funcionalidades en tiempo de ejecución.
  • Cuando la herencia no alcanza para extender el comportamiento.
  • Cuando querés respetar el principio abierto/cerrado (OCP).

Componentes principales 🧩

Component

Interfaz base que define la operación.

ConcreteComponent

Implementación principal de la interfaz.

Decorator

Implementa la interfaz y referencia al componente que envuelve.

ConcreteDecorators

Decoradores específicos que agregan comportamiento.

Component y Decorator base


// Component
class Notifier {
  send(message) { throw new Error("Debe implementarse"); }
}

// ConcreteComponent
class BasicNotifier extends Notifier {
  send(message) { console.log("Enviando:", message); }
}

// Decorator base: envuelve a otro Notifier y delega
class NotifierDecorator extends Notifier {
  constructor(notifier) { super(); this.notifier = notifier; }
  send(message) { this.notifier.send(message); }
}

ConcreteDecorators


class EmojiDecorator extends NotifierDecorator {
  send(message) { super.send("😀 " + message + " 🎉"); }
}

class TimestampDecorator extends NotifierDecorator {
  send(message) {
    const ts = new Date().toISOString();
    super.send(`[${ts}] ${message}`);
  }
}

Uso desde el cliente


// Composición dinámica: se apilan decoradores
let notifier = new BasicNotifier();
notifier = new TimestampDecorator(notifier);
notifier = new EmojiDecorator(notifier);

notifier.send("¡Hola mundo!");
// Enviando: 😀 [2025-06-24T12:00:00.000Z] ¡Hola mundo! 🎉

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • Extiende funcionalidad de forma flexible y dinámica.
  • Evita explosión de subclases por combinación.
  • Respeta la responsabilidad única.

⚠️ Desventajas

  • Puede generar muchos objetos anidados.
  • Más difícil de depurar que la herencia simple.

Ejemplos reales 🌍

☕ Cafetería de la UCU

Agrega ingredientes a una bebida de forma flexible, sin crear una clase por cada combinación posible.

Bebida base: café americano, latte.
Decoradores: leche, crema, jarabe de vainilla, canela, chocolate.

🔐 Sistema de autenticación

Suma controles sobre la autenticación base sin modificarla, respetando OCP.

Componente base: usuario autenticado.
Decoradores: doble factor, validación biométrica, captcha, firma digital, revisión de IP geográfica.

Ejemplo aplicado — editor de fotos 🖼️

Una ImagenOriginal se envuelve en capas apilables: Marco, FiltroSepia, Sticker, Texto… 


let foto = new ImagenOriginal();
foto = new Marco(foto);
foto = new FiltroSepia(foto);
foto = new Sticker(foto);
foto = new Texto(foto);

foto.Mostrar();
// "Imagen original + marco blanco + filtro sepia
//  + sticker de emoji 😎 + texto: 'Vacaciones 2025'"

Estructural

🏢 Facade

Una interfaz simple para un subsistema complejo.

¿Qué es?

🎯 Proporciona una interfaz simplificada a un conjunto complejo de clases o subsistemas, ocultando su complejidad al cliente.

¿Cuándo usarlo? 🧠

  • Cuando querés ocultar la complejidad de un sistema interno.
  • Cuando querés desacoplar al cliente del subsistema.
  • Cuando trabajás con librerías o APIs con muchas dependencias.

Subsistema 🎬


class Amplifier {
  on() { console.log("Amplificador encendido"); }
}

class StreamingPlayer {
  on() { console.log("Reproductor encendido"); }
  play(movie) { console.log(`Reproduciendo ${movie}`); }
}

class TheaterLights {
  dim(level) { console.log(`Luces atenuadas a ${level}%`); }

Facade: HomeTheaterFacade 🍿


class HomeTheaterFacade {
  constructor(amp, player, lights) {
    this.amp = amp; this.player = player; this.lights = lights;
  }

  // un solo método orquesta todo el subsistema
  watchMovie(movie) {
    console.log("Preparando para ver una película...");
    this.lights.dim(10);
    this.amp.on();
    this.player.on();
    this.player.play(movie);
  }
}

Sin Facade ❌ vs. con Facade ✅


// ❌ El cliente coordina cada paso, repetido en todos lados
lights.dim(10); amp.on(); player.on(); player.play("Matrix");

// ✅ El cliente delega en la fachada
const homeTheater = new HomeTheaterFacade(amp, player, lights);
homeTheater.watchMovie("Matrix");
// Preparando para ver una película...
// Luces atenuadas a 10% / Amplificador encendido / ...

Estructural

🛡️ Proxy

Un sustituto que controla el acceso a otro objeto.

¿Qué es?

🕵️ Provee un objeto sustituto que controla el acceso a otro objeto, pudiendo agregar lógica antes o después de delegar.

¿Cuándo usarlo? 🧠 y tipos 🔀

  • Para controlar el acceso a un objeto costoso o sensible.
  • Para agregar logs, control de acceso o caché sin tocar el objeto real.
  • Para cargar objetos de forma diferida (lazy loading).

Protección

Controla permisos de acceso.

Virtual

Crea objetos costosos bajo demanda.

Caché / Remoto

Reutiliza resultados o representa un objeto remoto.

Subject y Proxy virtual


// RealSubject: costoso de crear (carga al instanciarse)
class Video {
  constructor(filename) {
    this.filename = filename;
    this.load();
  }
  load() { console.log(`Cargando ${this.filename}...`); }
  play() { console.log(`Reproduciendo ${this.filename}`); }
}

// Proxy: difiere la carga hasta el primer play()
class VideoProxy {
  constructor(filename) { this.filename = filename; this.real = null; }
  play() {
    if (!this.real) this.real = new Video(this.filename); // lazy
    this.real.play();
  }
}

Uso desde el cliente


const video = new VideoProxy("pelicula.mp4");
console.log("Creado, pero todavía NO cargado");

video.play(); // recién acá se carga y reproduce
video.play(); // ya cargado → reproduce directamente

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • Controla el acceso a objetos costosos o sensibles.
  • Agrega logs, control de acceso o caché.

Tipos de Proxy 🔀

⏳ Virtual (init. diferida)

Difiere la creación de un objeto pesado hasta que se lo necesita de verdad.

🛡️ Protección (control de acceso)

Solo clientes autorizados pueden usar el objeto de servicio.

📝 Registro (logging)

Mantiene un historial de las solicitudes al objeto de servicio.

🧠 Referencia inteligente

Descarta el objeto pesado cuando ya no hay clientes que lo usen.

Ejemplo real 🌍 — Netflix / HBO Max / Disney+

Un Proxy actúa como filtro entre el usuario y el contenido: solo permite reproducir un video si el usuario…

  • está al día con el pago mensual del servicio;
  • tiene permiso para ese contenido según su tipo de suscripción;
  • está usando un dispositivo autorizado.
🎬 Combina control de acceso (protección) con inicialización diferida: el contenido pesado se carga recién al pasar los controles.

📦 Patrones de Persistencia

DAO · Repository

No forman parte del catálogo GoF, pero son patrones clásicos de acceso a datos: aíslan la lógica de persistencia del resto de la aplicación.

Persistencia

📂 DAO

Data Access Object — encapsula el acceso a datos.

¿Qué es?

📁 Encapsula el acceso a datos, separando la lógica de negocio de la lógica de persistencia.

¿Cuándo usarlo? 🧠

  • Cuando querés desacoplar la lógica de negocio del acceso a la base de datos.
  • Cuando querés facilitar el testeo o cambiar el mecanismo de persistencia.
  • Cuando tenés múltiples fuentes de datos o repositorios.

Estructura (UML)

Entidad e interfaz DAO


class User {
  constructor(id, name) { this.id = id; this.name = name; }
}

// Interfaz DAO: define el contrato de acceso a datos
class UserDAO {
  getAll()     { throw new Error("No implementado"); }
  getById(id)  { throw new Error("No implementado"); }
  save(user)   { throw new Error("No implementado"); }
}

Implementación concreta y uso


class InMemoryUserDAO extends UserDAO {
  constructor() { super(); this.users = []; }
  getAll()    { return this.users; }
  getById(id) { return this.users.find(u => u.id === id); }
  save(user)  { this.users.push(user); }
}

const dao = new InMemoryUserDAO();
dao.save(new User(1, "Alice"));
dao.save(new User(2, "Bob"));

console.log(dao.getById(2)); // User { id: 2, name: 'Bob' }

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • Separa el acceso a datos de la lógica de negocio.
  • Facilita el testeo y el mantenimiento.
  • Permite cambiar la persistencia sin tocar el resto.

⚠️ Desventajas

  • Puede generar clases repetitivas.
  • Innecesario en aplicaciones muy simples.
Persistencia

📦 Repository

Una colección del dominio sobre la fuente de datos.

¿Qué es?

📚 Actúa como una colección en memoria de objetos del dominio, ocultando los detalles de almacenamiento. Suele apoyarse en un DAO para acceder a los datos.

¿Cuándo usarlo? 🧠

  • Cuando querés abstraer la lógica de consulta y persistencia.
  • Cuando trabajás con modelos de dominio complejos.
  • Cuando querés desacoplar los servicios del almacenamiento.

Estructura (UML)

DAO de productos


class Product {
  constructor(id, name) { this.id = id; this.name = name; }
}

class ProductDAO {
  constructor() { this.storage = []; }
  getAll()    { return this.storage; }
  getById(id) { return this.storage.find(p => p.id === id); }
  save(p)     { this.storage.push(p); }
}

Repository sobre el DAO ✅


// El Repository ofrece una API del dominio y delega en el DAO
class ProductRepository {
  constructor(dao) { this.dao = dao; }
  getAllProducts()   { return this.dao.getAll(); }
  getProductById(id) { return this.dao.getById(id); }
  addProduct(p)      { this.dao.save(p); }
}

const repo = new ProductRepository(new ProductDAO());
repo.addProduct(new Product(1, "Lámpara"));
repo.addProduct(new Product(2, "Sofá"));

console.log(repo.getProductById(1)); // Product { id: 1, name: 'Lámpara' }

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • Encapsula la lógica de acceso y consulta.
  • Ofrece una API coherente sobre el dominio.
  • Facilita pruebas y mantenimiento.

⚠️ Desventajas

  • Puede duplicar trabajo si ya hay un ORM sofisticado.
  • Agrega una capa que puede sobrar en sistemas simples.

🆚 DAO vs. Repository

📂 DAO📦 Repository
PropósitoCRUD directoColecciones y reglas de dominio
NivelInferior, cercano al almacenamientoSuperior, cercano a los objetos de negocio
FocoPersistencia técnica (oculta SQL, JDBC, Hibernate)El dominio, no los detalles técnicos
MetáforaSecretario técnico de la base de datosBibliotecario

🔄 Patrones de Comportamiento

CoR · Command · Observer · Mediator · Memento · State · Strategy · Visitor

Los patrones de comportamiento describen cómo interactúan los objetos y cómo se reparten las responsabilidades, mejorando la comunicación y reduciendo el acoplamiento.

Comportamiento

⛓️ Chain of Responsibility

Pasá la solicitud por una cadena de manejadores.

¿Qué es?

Permite que varios objetos tengan la oportunidad de manejar una solicitud. Los manejadores se encadenan y la solicitud se pasa por la cadena hasta que uno la procesa, sin que el emisor sepa quién lo hará.

¿Cuándo usarlo? 🧠 y componentes 🧩

  • Cuando querés desacoplar al emisor de los receptores.
  • Cuando más de un objeto puede manejar una solicitud.
  • Cuando querés agregar manejadores sin modificar el código existente.

Handler

Define el contrato y la referencia al siguiente.

ConcreteHandler

Maneja o delega al siguiente.

Client

Crea y configura la cadena.

Handler base y concretos


class Handler {
  setNext(handler) { this.next = handler; return handler; }
  handle(request) {
    return this.next ? this.next.handle(request) : null;
  }
}

class SupportHandler extends Handler {
  handle(request) {
    if (request.type === "soporte") return "🎧 Soporte respondió";
    return super.handle(request);
  }
}

class BillingHandler extends Handler {
  handle(request) {
    if (request.type === "factura") return "💰 Facturación procesó";
    return super.handle(request);
  }
}

class GeneralHandler extends Handler {
  handle(request) { return "📨 Derivado a atención general"; }
}

Uso desde el cliente


const soporte = new SupportHandler();
const facturacion = new BillingHandler();
const general = new GeneralHandler();

soporte.setNext(facturacion).setNext(general);

console.log(soporte.handle({ type: "soporte" })); // 🎧 Soporte respondió
console.log(soporte.handle({ type: "factura" })); // 💰 Facturación procesó
console.log(soporte.handle({ type: "otro" }));    // 📨 Atención general

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • Desacopla el emisor del receptor.
  • Fácil de extender con nuevos manejadores.
  • Permite controlar el flujo dinámicamente.

⚠️ Desventajas

  • Difícil de seguir si la cadena es muy larga.
  • No garantiza que alguien maneje la solicitud.
Comportamiento

🎮 Command

Encapsulá una solicitud como un objeto.

¿Qué es?

Encapsula una solicitud como un objeto, permitiendo parametrizar acciones, encolarlas, mantener un historial y soportar operaciones de deshacer / rehacer.

Componentes principales 🧩

Command

Interfaz con execute().

ConcreteCommand

Implementa execute() y llama al receptor.

Receiver

Hace el trabajo real.

Invoker

Dispara el comando.

Client

Configura todo.

Receiver y Commands


class Light {
  on()  { console.log("💡 Luz encendida"); }
  off() { console.log("🌑 Luz apagada"); }
}

class Command { execute() {} }

class LightOnCommand extends Command {
  constructor(light) { super(); this.light = light; }
  execute() { this.light.on(); }
}

class LightOffCommand extends Command {
  constructor(light) { super(); this.light = light; }
  execute() { this.light.off(); }
}

Invoker y uso


class RemoteControl {
  setCommand(command) { this.command = command; }
  pressButton() { this.command.execute(); }
}

const light = new Light();
const remote = new RemoteControl();

remote.setCommand(new LightOnCommand(light));
remote.pressButton(); // 💡 Luz encendida

remote.setCommand(new LightOffCommand(light));
remote.pressButton(); // 🌑 Luz apagada

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • Desacopla el invocador del receptor.
  • Permite historial, undo/redo y encolar comandos.

⚠️ Desventajas

  • Aumenta la cantidad de clases.
  • Excesivo si solo se necesita una simple llamada.

Comportamiento

👀 Observer

Notificá automáticamente a quienes dependen de un objeto.

¿Qué es?

Define una dependencia uno-a-muchos: cuando un objeto cambia de estado, todos sus dependientes son notificados automáticamente.

Componentes principales 🧩

Subject

Mantiene la lista de observadores y los notifica.

Observer

Interfaz con el método update().

ConcreteObserver

Implementa update() y reacciona.

Subject y Observer


class Subject {
  constructor() { this.observers = []; }
  subscribe(o)   { this.observers.push(o); }
  unsubscribe(o) { this.observers = this.observers.filter(x => x !== o); }
  notify(data)   { this.observers.forEach(o => o.update(data)); }
}

class Observer {
  update(data) { throw new Error("Debe implementarse"); }
}

ConcreteObservers y uso


class LoggerObserver extends Observer {
  update(data) { console.log("📝 Log:", data); }
}
class AlertObserver extends Observer {
  update(data) { console.log("🔔 Alerta:", data); }
}

const subject = new Subject();
subject.subscribe(new LoggerObserver());
subject.subscribe(new AlertObserver());

subject.notify("Temperatura crítica");
// 📝 Log: Temperatura crítica
// 🔔 Alerta: Temperatura crítica

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • Desacopla al sujeto de sus observadores.
  • Permite agregar observadores en tiempo de ejecución.
  • Ideal para eventos y notificaciones.

⚠️ Desventajas

  • Difícil de depurar con muchos observadores.
  • El orden de notificación no está garantizado.

🧩 Mediator

Centralizá la comunicación entre objetos.

¿Qué es?

Centraliza la comunicación entre objetos en un mediador. En lugar de referenciarse entre sí, los objetos se comunican a través de él, reduciendo el acoplamiento.

Componentes principales 🧩

Mediator

Interfaz de comunicación entre colegas.

ConcreteMediator

Implementa la lógica de coordinación.

Colleague

Interfaz base de los objetos que interactúan.

ConcreteColleague

Se comunica a través del mediador.

Colleagues


class Colleague {
  constructor(mediator) { this.mediator = mediator; }
  receive(message) {}
}

class Button extends Colleague {
  click() {
    console.log("🔘 Botón clickeado");
    this.mediator.notify(this, "click");
  }
}

class TextBox extends Colleague {
  receive(message) { console.log("📥 TextBox:", message); }
}

Mediator concreto y uso


class FormMediator {
  registerButton(button)   { this.button = button; }
  registerTextBox(textBox) { this.textBox = textBox; }
  notify(sender, event) {
    if (sender === this.button && event === "click") {
      this.textBox.receive("Formulario enviado");
    }
  }
}

const mediator = new FormMediator();
const button = new Button(mediator);
const textBox = new TextBox(mediator);
mediator.registerButton(button);
mediator.registerTextBox(textBox);

button.click();
// 🔘 Botón clickeado
// 📥 TextBox: Formulario enviado

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • Reduce dependencias entre componentes.
  • Centraliza la lógica de interacción.
  • Facilita el mantenimiento y las pruebas.

⚠️ Desventajas

  • El mediador puede volverse muy complejo.
  • Añade una capa de indirección.
Comportamiento

🧠 Memento

Guardá y restaurá el estado sin romper el encapsulamiento.

¿Qué es?

Captura y almacena el estado interno de un objeto sin violar su encapsulamiento, de modo que pueda restaurarse más tarde.

Componentes principales 🧩

Originator

El objeto cuyo estado se quiere guardar.

Memento

Almacena el estado del originador.

Caretaker

Guarda y restaura los mementos.

Originator, Memento y Caretaker


class Memento {
  constructor(state) { this.state = state; }
  getState() { return this.state; }
}

class Editor {
  constructor() { this.content = ""; }
  write(text) { this.content += text; }
  save()      { return new Memento(this.content); }
  restore(m)  { this.content = m.getState(); }
}

class History {
  constructor() { this.stack = []; }
  push(m) { this.stack.push(m); }
  pop()   { return this.stack.pop(); }
}

Uso desde el cliente


const editor = new Editor();
const history = new History();

editor.write("Hola ");
history.push(editor.save()); // checkpoint

editor.write("Mundo!");
console.log(editor.content); // Hola Mundo!

editor.restore(history.pop());
console.log(editor.content); // Hola

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • No rompe el encapsulamiento.
  • Permite implementar “deshacer” fácilmente.
  • Útil en editores, juegos y formularios.

⚠️ Desventajas

  • Puede consumir mucha memoria con muchos estados.
  • El caretaker debe gestionar el ciclo de vida.
Comportamiento

🔁 State

El objeto cambia de comportamiento al cambiar de estado.

¿Qué es?

Permite que un objeto cambie su comportamiento cuando cambia su estado interno: el objeto parece cambiar de clase.

¿Cuándo usarlo? 🧠 y componentes 🧩

  • Cuando el comportamiento depende del estado del objeto.
  • Cuando hay muchos if/else o switch sobre estados.

Context

Mantiene una referencia al estado actual.

State

Interfaz común a todos los estados.

ConcreteState

Comportamiento concreto de cada estado.

States y Context


class State {
  handle(context) { throw new Error("Debe implementarse"); }
}

class HappyState extends State {
  handle(context) {
    console.log("😀 Estoy feliz!");
    context.setState(new SadState());
  }
}
class SadState extends State {
  handle(context) {
    console.log("😢 Estoy triste...");
    context.setState(new HappyState());
  }
}

class Person {
  constructor() { this.state = new HappyState(); }
  setState(state) { this.state = state; }
  react() { this.state.handle(this); }
}

const person = new Person();

person.react(); // 😢 Estoy triste...

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • Elimina estructuras de control repetitivas.
  • Encapsula el comportamiento de cada estado.
  • Facilita agregar nuevos estados.

⚠️ Desventajas

  • Aumenta la cantidad de clases.
  • Innecesario con pocos estados simples.

Algoritmos intercambiables encapsulados.

¿Qué es?

Define una familia de algoritmos, los encapsula y los hace intercambiables, permitiendo que el algoritmo varíe de forma independiente del cliente que lo usa.

Componentes principales 🧩

Strategy

Interfaz común a las estrategias.

ConcreteStrategy

Implementaciones concretas del algoritmo.

Usa una estrategia para ejecutar el algoritmo.

class PaymentStrategy {
  pay(amount) { throw new Error("Debe implementarse"); }
}

}

  pay(amount) { console.log(`📧 $${amount} con PayPal`); }

class PaymentContext {
  constructor(strategy) { this.strategy = strategy; }

const pago = new PaymentContext(new CreditCardPayment());

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • Elimina estructuras de control complejas.
  • Fácil de extender con nuevas estrategias.

⚠️ Desventajas

  • Aumenta el número de clases.
  • El cliente debe conocer las diferencias entre estrategias.
Comportamiento

🧭 Visitor

Agregá operaciones sin modificar las clases.

¿Qué es?

Permite agregar nuevas operaciones a una estructura de objetos existente sin modificar sus clases, separando el algoritmo de los datos.

Componentes principales 🧩

Visitor

Declara una operación por cada tipo de elemento.

ConcreteVisitor

Implementa esas operaciones.

Element

Define accept(visitor).

ConcreteElement

Llama a visitor.visit(this).

Visitor y Elements


class Visitor {
  visitCircle(circle) {}
  visitRectangle(rectangle) {}
}

class Shape { accept(visitor) { throw new Error("Debe implementarse"); } }

class Circle extends Shape {
  constructor(radius) { super(); this.radius = radius; }
  accept(visitor) { visitor.visitCircle(this); }
}
class Rectangle extends Shape {
  constructor(w, h) { super(); this.width = w; this.height = h; }
  accept(visitor) { visitor.visitRectangle(this); }
}

ConcreteVisitor y uso


class AreaCalculator extends Visitor {
  visitCircle(c)    { console.log(`🔵 ${(Math.PI * c.radius ** 2).toFixed(2)}`); }
  visitRectangle(r) { console.log(`🟦 ${r.width * r.height}`); }
}

const shapes = [new Circle(5), new Rectangle(4, 6)];
const visitor = new AreaCalculator();

shapes.forEach(shape => shape.accept(visitor));
// 🔵 78.54
// 🟦 24

✅ Ventajas y ⚠️ Desventajas

✅ Ventajas

  • Agrega operaciones sin modificar las clases existentes.
  • Fomenta la responsabilidad única.
  • Facilita mantener algoritmos complejos.

⚠️ Desventajas

  • Agregar un elemento obliga a tocar todos los visitantes.
  • Rompe OCP si la jerarquía cambia seguido.
